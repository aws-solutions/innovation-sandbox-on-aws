# Lease extension request/approval/denial workflow

This change adds a three-phase lease extension lifecycle: users request extensions, admins review them, and both sides receive email notifications. The implementation stores a `pendingExtensionRequest` on the `MonitoredLease` document, enforces max-duration validation server-side, wires two new API routes through the existing leases Lambda, and adds frontend UI for requesting and approving/denying extensions with a tab-based split in the approvals list.

Watch for: no guard against duplicate extension requests overwriting a pending one (confirmed); no server-side validation that the requested date is actually in the future or past the current expiration (confirmed); domain logic methods lack unit tests â€” all testing is at the handler level via mocked `InnovationSandbox` calls (confirmed).

## High-level view

The domain layer adds three static methods to `InnovationSandbox` that follow the existing event-sourced pattern: mutate lease state, emit an EventBridge event, log. The max-duration check computes hours from `startDate` to the requested expiration, rejecting requests that exceed `globalConfig.leases.maxDurationHours`. No check ensures the request actually extends the lease â€” a user could "extend" to a date earlier than the current expiration and it would succeed.

The API layer adds `/leases/{leaseId}/extend` for users and `/leases/{leaseId}/extend/review` for admins. The review endpoint reads the `requestedExpirationDate` from the stored `pendingExtensionRequest`, not from the request body, which prevents an admin from approving a different date than what was requested. However, nothing prevents an Admin from requesting and then immediately approving their own extension.

The frontend adds a request modal with client-side validation (date-in-future, max-duration), a dedicated extension approval details page, and a second tab in the approvals list for extension requests. The `useGetPendingExtensions` hook filters the full leases list client-side rather than making a dedicated API call.

## Pending extension request state and overwrite risk

The `requestLeaseExtension` method unconditionally writes a new `pendingExtensionRequest` onto the lease. If a user submits a second request before the first is reviewed, the original request (with its `requestedAt` timestamp and potentially different expiration date) is silently replaced. The admin notification for the first request still links to the same lease, but the stored request will have changed by the time they review it.

There's also no optimistic concurrency on this write â€” the `update` call omits the `expected` parameter, so if the lease is modified between the handler's read and the domain method's write, the entire lease state is overwritten with the stale snapshot plus the new `pendingExtensionRequest`. Given that two actors (user and admin) may be operating on the same lease concurrently (user re-requesting while admin is reviewing), this is a realistic race.

## Missing server-side validation for the requested date

The domain logic validates that the requested date does not exceed `maxDurationHours` from `startDate`. It does not validate:

- Requested date > now
- Requested date > current expiration date

If a request for a past date is submitted and approved, the lease's `expirationDate` would be set backwards, potentially triggering immediate expiration on the next lease monitor cycle.

The relevant section in `innovation-sandbox.ts`:
```typescript
const durationInHours = requestedExpiration.diff(startDate, "hours").hours;
if (durationInHours > globalConfig.leases.maxDurationHours) {
  throw new LeaseExtensionExceedsMaxDurationError(...);
}
```

No check for `requestedExpiration > now()` or `requestedExpiration > parseDatetime(lease.expirationDate)`.

## Self-approval gap

The review endpoint gates on Admin/Manager role via `authorizedToLeaseFromPrivateLeaseTemplates`. The request endpoint allows Admin users to request extensions on their own leases (they pass `isUserNotAllowedByEmail`). The review endpoint does not check whether the approver is the same person who requested the extension. An Admin could request and immediately approve their own extension without any other human in the loop.

## Test coverage gaps

Not tested:
- Domain logic (`InnovationSandbox.requestLeaseExtension` etc.) directly â€” all handler tests mock the domain methods, so the max-duration validation logic, event emission, and lease store interaction are never exercised end-to-end.
- Concurrent request/review scenario.
- Frontend extension components (the `LeaseDetails.test.tsx` change only adds a `useModal` mock to fix the existing test).

## Frontend comment-length discrepancy

The backend `RequestLeaseExtensionBodySchema` allows comments up to 1000 characters, but the frontend's `RequestExtensionModal` validation schema limits comments to 500 characters. This means the client will reject valid requests that the server would accept.

- Handler: `comments: z.string().max(1000).optional()`
- Frontend: `comments: z.string().max(500).optional()`
